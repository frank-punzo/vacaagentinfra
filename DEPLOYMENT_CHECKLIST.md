# VacaAgent Terraform Deployment Checklist

This document provides a complete checklist for destroying and rebuilding the VacaAgent infrastructure using Terraform.

## Prerequisites

### 1. Required Files in `vacaagentinfra` Directory

Verify these files exist and are up-to-date:

- ✅ `main.tf` - Main Terraform configuration
- ✅ `variables.tf` - Variable definitions (includes `lambda_log_level`)
- ✅ `outputs.tf` - Output definitions
- ✅ `lambda.tf` - Lambda function and layer configuration (uses archive_file data source)
- ✅ `iam.tf` - IAM roles and policies
- ✅ `vpc.tf` - VPC and networking configuration
- ✅ `rds.tf` - RDS database configuration
- ✅ `s3.tf` - S3 bucket configuration
- ✅ `cognito.tf` - Cognito user pool configuration
- ✅ `api_gateway.tf` - API Gateway configuration
- ✅ `security_groups.tf` - Security group definitions
- ✅ `secrets.tf` - Secrets Manager configuration
- ✅ `lambda_layer.zip` - Lambda dependencies layer
- ⚠️ `lambda_function.zip` - AUTO-GENERATED by Terraform (no manual updates needed)

### 2. Lambda Source Code Location

**AUTOMATIC DEPLOYMENT:** Terraform now automatically packages Lambda code from source!

The `archive_file` data source in `lambda.tf` reads from:
```
../vacaagent/lambda/src
```

**No manual zip creation needed!** Terraform will:
1. Automatically zip the contents of `lambda/src` directory
2. Detect any changes to source files
3. Update Lambda function when source code changes
4. Output the zip to `vacaagentinfra/lambda_function.zip`

Expected source files in `lambda/src`:
- ✅ `index.py` - Main handler with logging configuration

## Deployment Steps

### Step 1: Verify Lambda Source Code

```bash
# Navigate to Lambda source directory
cd /c/homestuff/code/vacaagent/lambda/src

# Verify all files are present
ls -la controllers/
ls -la utils/
ls -la index.py

# No manual zip creation needed - Terraform handles this automatically!
```

### Step 2: Initialize Terraform

```bash
cd /c/homestuff/code/vacaagentinfra

# Initialize Terraform (downloads providers)
terraform init
```

### Step 3: Review Current State (if exists)

```bash
# View current resources
terraform show

# List all resources
terraform state list
```

### Step 4: Destroy Existing Infrastructure

**WARNING:** This will delete all data in RDS and S3!

```bash
# Review what will be destroyed
terraform plan -destroy

# Destroy infrastructure (requires confirmation)
terraform destroy

# Or auto-approve (use with caution)
terraform destroy -auto-approve
```

### Step 5: Clean State (if needed)

```bash
# If destroy fails or you need to start fresh
rm -f terraform.tfstate
rm -f terraform.tfstate.backup
rm -rf .terraform/
```

### Step 6: Deploy Fresh Infrastructure

```bash
# Review deployment plan
terraform plan

# Apply infrastructure
terraform apply

# Or auto-approve
terraform apply -auto-approve
```

### Step 7: Verify Deployment

```bash
# Check outputs
terraform output

# Should show:
# - api_gateway_url
# - cognito_user_pool_id
# - cognito_user_pool_client_id
# - s3_photos_bucket
# - vpc_id
```

### Step 8: Initialize Database Schema

**IMPORTANT:** After infrastructure is deployed, initialize the database:

```bash
# Connect to RDS from a bastion host or EC2 instance in the VPC
# Get RDS endpoint from Terraform output
RDS_ENDPOINT=$(terraform output -raw rds_endpoint)

# Get DB password from Secrets Manager
DB_PASSWORD=$(aws secretsmanager get-secret-value --secret-id vacaagent-db-password --query SecretString --output text | jq -r .password)

# Connect to database
psql -h $RDS_ENDPOINT -U vacaadmin -d vacaagent

# Run schema initialization (from lambda/schema.sql)
\i /path/to/vacaagent/lambda/schema.sql
```

### Step 9: Update Frontend Configuration

After deployment, update the React Native app configuration:

**File:** `src/config/aws-config.js`

```javascript
export const AWS_CONFIG = {
  cognito: {
    userPoolId: '<terraform output cognito_user_pool_id>',
    userPoolClientId: '<terraform output cognito_user_pool_client_id>',
    region: 'us-east-1',
  },
  api: {
    endpoint: '<terraform output api_gateway_url>',
  },
};
```

### Step 10: Test Deployment

```bash
# Test API Gateway endpoint
API_URL=$(terraform output -raw api_gateway_url)
curl ${API_URL}/health

# Test Lambda function directly (if health endpoint exists)
aws lambda invoke --function-name vacaagent-api --payload '{}' response.json

```

## Post-Deployment Verification

### Check CloudWatch Logs

```bash
# View Lambda logs
aws logs tail /aws/lambda/vacaagent-api --follow

# Test logging levels
aws lambda update-function-configuration \
  --function-name vacaagent-api \
  --environment "Variables={LOGLEVEL=DEBUG,...}"
```

### Verify IAM Permissions

Lambda should have permissions for:
- ✅ CloudWatch Logs (create log groups, streams, put events)
- ✅ VPC Access (create/delete network interfaces)
- ✅ Secrets Manager (get secret values)
- ✅ S3 (read/write photos bucket)
- ✅ Cognito (admin user operations)

### Verify Security Groups

- ✅ Lambda security group allows outbound to RDS port 5432
- ✅ RDS security group allows inbound from Lambda security group
- ✅ Lambda security group allows outbound HTTPS for Secrets Manager

### Verify VPC Configuration

- ✅ Lambda runs in private subnets
- ✅ RDS runs in private subnets (at least 2 AZs)
- ✅ VPC endpoints exist for Secrets Manager (or NAT Gateway)

## Troubleshooting

### Issue: Lambda Cannot Connect to RDS

**Symptoms:**
- Timeout errors in Lambda logs
- "Unable to connect to database" errors

**Solutions:**
1. Verify Lambda and RDS are in same VPC
2. Check security group rules (Lambda SG → RDS SG on port 5432)
3. Verify RDS is in private subnets
4. Check subnet route tables have proper routes

### Issue: Lambda Cannot Access Secrets Manager

**Symptoms:**
- "Unable to retrieve secret" errors
- Timeout accessing Secrets Manager

**Solutions:**
1. Verify VPC endpoint for Secrets Manager exists
2. If no endpoint, ensure Lambda has internet access (NAT Gateway)
3. Check IAM role permissions
4. Verify security group allows outbound HTTPS

### Issue: API Gateway Returns 502 Bad Gateway

**Symptoms:**
- 502 errors from API Gateway
- No logs in CloudWatch

**Solutions:**
1. Check Lambda function permissions for API Gateway
2. Verify Lambda function exists and is not in failed state
3. Check Lambda timeout settings
4. Review Lambda logs for errors

### Issue: Frontend Cannot Authenticate

**Symptoms:**
- "User pool does not exist" errors
- Authentication failures

**Solutions:**
1. Verify Cognito User Pool ID and Client ID in frontend config
2. Check Cognito User Pool has app client configured
3. Verify app client settings (no secret, SRP auth flow)
4. Check CORS settings on API Gateway

## Important Notes

### Lambda Deployment Package

**AUTOMATIC UPDATES:** Terraform now automatically detects and deploys Lambda code changes!

The `archive_file` data source monitors the `lambda/src` directory:
- Detects any file changes automatically
- Calculates new source_code_hash
- Triggers Lambda function update on `terraform apply`

**To update Lambda code after infrastructure is deployed:**

```bash
# Simply make changes to source files, then:
cd /c/homestuff/code/vacaagentinfra
terraform plan   # Shows Lambda function will be updated
terraform apply  # Deploys the changes

# No manual zip creation or AWS CLI commands needed!
```

**How it works:**
1. Edit files in `vacaagent/lambda/src/`
2. Run `terraform apply`
3. Terraform automatically zips source and updates Lambda

### Database Backups

**IMPORTANT:** RDS has automated backups enabled, but they are deleted on destroy!

Before destroying infrastructure, consider:
1. Creating a manual snapshot
2. Exporting data
3. Backing up critical information

```bash
# Create manual snapshot before destroy
aws rds create-db-snapshot \
  --db-instance-identifier vacaagent-db \
  --db-snapshot-identifier vacaagent-backup-$(date +%Y%m%d)
```

### State Management

Terraform state is stored locally by default. For team collaboration:
1. Consider using remote state (S3 + DynamoDB)
2. Never commit `terraform.tfstate` to git
3. Keep backups of state files

## Environment Variables Summary

Lambda receives these environment variables from Terraform:

| Variable | Source | Purpose |
|----------|--------|---------|
| `DB_SECRET_ARN` | Terraform | RDS password from Secrets Manager |
| `PHOTOS_BUCKET` | Terraform | S3 bucket for photos |
| `USER_POOL_ID` | Terraform | Cognito User Pool ID |
| `USER_POOL_CLIENT_ID` | Terraform | Cognito App Client ID |
| `ENVIRONMENT` | Terraform variable | Environment name (dev/prod) |
| `LOGLEVEL` | Terraform variable | Python logging level |
